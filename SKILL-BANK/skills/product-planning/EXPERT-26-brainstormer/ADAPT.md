# Adapting Brainstormer

**What this skill does:** turns a raw idea into a structured, vision-ready brainstorm document
through Socratic questioning, assumption surfacing, contradiction resolution, and portfolio
gap analysis.

This skill is generic methodology — the five-phase questioning workflow works for any
organization and never hardcodes a specific business. To use it for your organization, decide
the values below and either keep them in your brainstorm brief at use time, or bake them into
your forked copy of `SKILL.md`. Until they're decided, the skill emits a labeled GENERIC DRAFT
(classification, portfolio-overlap, and lifecycle notes marked `PLACEHOLDER`) rather than
inventing your business model or portfolio.

## What to decide for your organization

| Decide | Used in `SKILL.md` under | Notes |
|--------|--------------------------|-------|
| **Business model** — one line for how the organization makes money | Brainstorming philosophy → Business-Model Lens; Handoff protocol | Drives the product-vs-service classification. |
| **Offering taxonomy** — products vs. client services, or your own categories | Business-Model Lens; Product-vs-Service Classification | If you name other offering types, classify against those. |
| **Existing portfolio** — the offerings that already exist (name + lifecycle stage) | Portfolio Awareness; Phase 0 portfolio check | Used to detect overlap; propose extending vs. creating new. |
| **Lifecycle stages** — your stage names, or the generic default | Lifecycle Placement; Phase 4 readiness; Handoff | Default: Vision → Feasibility → Design & Planning → Development → Launch & Growth. |
| **Core differentiating capability** — what you uniquely enable (e.g. AI-native) | 3-Lens Expansion (AI-native lens) | Becomes the third expansion lens. |
| **Audience / ICP segments** — target customer segments | Phase 1 WHO questions (persona expansion) | Anchor persona expansion in real segments, not invented ones. |
| **Product-folder location** — where an offering's docs live + template/file names | Phase 4 write step; Cross-skill integration; `forms.md` context sources | Where the finished brainstorm document is written. |
| **Organization / offering name** — for framing and deriving the offering code | `forms.md` vision-ready marker (`<!-- VISION_READY: CODE -->`) | Code can be derived from the offering name if not supplied. |

## How to gather these

The numbered **Org Context Intake** in [`forms.md`](forms.md) is the questionnaire for all of
the above — ask it once, record the answers wherever your team keeps these decisions.

## Platform-independent (do not change)

The questioning method is universal and needs no adaptation: the five-phase workflow, the Hard
Questions Framework (WHY / WHO / WHAT / HOW / WHAT-IF), the contradiction & vagueness detection
patterns and Vague Term Dictionary in [`reference.md`](reference.md), and the brainstorm
document template, quality checklists, and activation checklist in [`forms.md`](forms.md) all
work as-is for any organization.
