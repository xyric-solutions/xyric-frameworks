# Adapting Presentation System

**What this skill does:** builds and reviews on-brand presentation decks that make an
argument — applying an organization's color/type system, logo, and positioning to a
claim-driven slide narrative.

This skill is generic methodology — it never hardcodes a specific brand or message. To use
it for your organization, decide the values below and either keep them in your deck brief at
use time, or bake them into your forked copy of `SKILL.md`. Until they're decided, the skill
emits a labeled GENERIC DRAFT (system fonts, neutral colors) rather than inventing a brand or
positioning.

## What to decide for your organization

| Decide | Used in `SKILL.md` under | Notes |
|--------|--------------------------|-------|
| **Palette** — primary, accent, light base, dark base (name + hex each) | Core rules (color system), Anti-patterns | Note which is "use sparingly" (usually the accent). |
| **Typefaces** — display/headline font, technical/mono font for labels & data | Core rules (type system) | State a system-font fallback. |
| **Logo / wordmark** — exact wordmark string (incl. punctuation), asset location, light-on-dark vs dark-on-light variants | Core rules (wordmark), Workflow step 4 | Used on cover, close, and sign-off. |
| **Tagline / sign-off** — exact string incl. punctuation | Core rules (tagline) | Reserved for the close; never paraphrase. |
| **Structural visual device** — signature motif (grid, connected nodes, stepped band) that recurs | Core rules (structural device), Anti-patterns | Must encode a real relationship, not decoration. |
| **Core promise** — the one value statement the deck must land | Deliverable standard, Core rules (tension→resolution) | The close restates this. |
| **Key messages** — 3-7 claims the deck must make | Core rules (claim spine), Workflow step 3 | One claim per content slide. |
| **Differentiators** — what makes the offer different from the obvious alternative | Core rules, Anti-patterns | Drives the proof objects. |
| **Template deck** — reusable starter `.pptx` and its layout map, if one exists | Workflow steps 2 & 5, `reference.md` Layout Selection | Reuse its layouts before inventing new ones. |

## How to gather these

The **Deck Intake** in [`forms.md`](forms.md) is the questionnaire for all of the above —
ask it once, record the answers wherever your team keeps brand and positioning decisions.

## Platform-independent (do not change)

`forms.md` (Claim Spine planner, Presentation QA checklist, copy patterns) and `reference.md`
(layout selection, visual-mode system, source order, slide rhythm, asset hierarchy) are
deck-craft facts that work as-is for any organization — only the brand and positioning values
you supply change.
