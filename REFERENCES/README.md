# References

Use this folder for reusable decision guides, standards, and reference documents that help humans and AI agents choose the right approach.

References are not skills and they are not full project frameworks. They should be read selectively when a task needs their domain context.

## What Belongs Here

- Technology stack guidance.
- Engineering standards and reusable decision records.
- Portable checklists for choosing tools, patterns, or implementation approaches.
- Source links and migration notes for older framework documents.

## What Does Not Belong Here

- Client-specific material, secrets, private paths, or personal setup notes.
- Central org configuration systems or profile databases.
- Long tutorials that should live in product docs.
- Installable AI capabilities, which belong in `SKILL-BANK/`.

## Available References

| Reference | Use it when |
|-----------|-------------|
| `technology/Xyric-Tech-Stack.md` | Choosing or reviewing default tools for a Xyric software project. |
| `technology/stacks/` | Choosing a reusable implementation pattern for web, mobile, or backend services. |

## How Agents Should Use References

1. Read only the reference that matches the current task.
2. Treat default decisions as the starting point.
3. Use approved alternatives only when the project context justifies them.
4. If a decision is missing or marked as needing review, ask for the decision or record it as unresolved.
5. Do not invent organization-specific policy, tooling, vendors, or compliance requirements.
